# KAIROS v0.1 — Implementation Audit

Conducted against the current product contract (README.md, docs/architecture.md,
docs/cli.md, docs/v0.1-status.md, CONTRIBUTING.md) and the historical planning
document (docs/implementation-plan.md). This document was written before any
remediation code was touched. Every "verified" status below is backed by an
existing automated test or a command whose output was inspected; nothing is
marked verified on the basis of "the code appears to do this."

Status vocabulary: **verified** (automated evidence exists and passes today),
**partially verified** (some evidence exists but a gap remains), **violated**
(the contract claim is false as written), **undocumented implementation**
(code does something the docs don't claim, or vice versa — a mismatch, not
necessarily a bug), **not yet testable** (no evidence either way).

## 1. Offline and local-only guarantee

| # | Claim | Source | Status |
|---|---|---|---|
| 1.1 | "No cloud dependency, no telemetry... no network access" | README.md "Why KAIROS"; architecture.md storage layout | **partially verified → remediated** |
| 1.2 | "It is fully usable without any LLM, requires no network access" | README.md intro | **partially verified → remediated** |

**Evidence before remediation:** `grep` across `src/` for `requests`, `httpx`,
`urllib`, `socket`, `http.client`, `ftplib`, `smtplib`, `aiohttp` — zero
matches. The six direct dependencies (typer, rich, pydantic, sqlalchemy,
alembic, pypdf) are well-known offline libraries with no update-check or
telemetry behavior. This is consistent with the claim but was never
*enforced* — nothing would fail loudly if a future change (or a transitive
dependency upgrade) introduced a network call.

**One legitimate exception, already documented, now explicitly verified
non-network:** `kairos.infrastructure.git.metadata.read_git_metadata` shells
out to `git rev-parse`, `git rev-parse --abbrev-ref HEAD`, `git remote
get-url origin` — three read-only, local-config-only git subcommands. None
of them touch the network (verified by inspecting each command: `rev-parse`
reads local refs; `remote get-url` reads `.git/config`; neither is
`fetch`/`pull`/`clone`/`ls-remote`). This is the sole `subprocess` call in
the codebase.

**Remediation (smallest necessary):** Added
`tests/integration/test_offline.py`, which monkeypatches `socket.socket`,
`socket.create_connection`, and `socket.getaddrinfo` to raise
`AssertionError` on any call, then drives the full CLI workflow (init,
ingest all six fixture kinds, artifacts, show, search, trace, config, logs,
well, note, doctor) through `CliRunner` in-process. This is the "monkeypatch
common network entry points" option the audit calls for — cheaper and more
portable in CI than OS-level network denial, and it exercises the exact
Python-level call sites that would need to change for KAIROS to ever violate
this guarantee.

**Verification command:**
```
pytest tests/integration/test_offline.py -v
```

**Residual risk:** the git-metadata subprocess call is not covered by the
socket-patching test (it's a separate process). It is verified non-network
by code inspection of the three git subcommands used, not by runtime
sandboxing. Documented here rather than sandboxed further, since adding
process-level network denial (e.g. `unshare -n` or firewall rules) is
heavier CI infrastructure than this milestone's risk warrants — the git
commands used are read-only local-config commands by construction, and
`read_git_metadata` already treats any failure (including a network-capable
git config doing something unexpected) as non-fatal, degrading to `None`.

## 2. Source immutability and content-addressed raw storage

| # | Claim | Source | Status |
|---|---|---|---|
| 2.1 | "KAIROS is read-only with respect to registered source material" | architecture.md "Read-only boundary" | **verified** |
| 2.2 | Raw bytes stored content-addressed, write-once, read-only chmod | architecture.md storage layout; `content_store.py` | **verified** (existing unit tests) |
| 2.3 | Repeated ingestion of identical bytes is idempotent | cli.md "already ingested" | **verified** (existing integration test) |
| 2.4 | Raw-byte integrity is checkable | audit requirement (not previously an explicit product claim) | **violated → remediated** |

**Evidence:** `ContentStore.put()` only ever opens the *source* path for
reading (`source_path.open("rb")`) — never for writing. The destination in
`.kairos/content/<sha256[:2]>/<sha256>` is only written if it doesn't
already exist, then chmodded `0o444`. `test_content_store.py` already proves
write-once (`test_put_is_idempotent_and_never_overwrites` — same mtime after
a second `put()` of identical bytes) and content-addressing
(`test_different_content_gets_different_hash`). `test_ingest_is_idempotent_on_same_content`
proves the CLI-level idempotency ("already ingested").

Note on the storage path: the audit brief assumed
`<workspace>/.kairos/raw/<sha256>`. The actual, already-documented path is
`<workspace>/.kairos/content/<sha256[:2]>/<sha256>` (sharded by the first two
hex digits, to avoid one directory holding thousands of files). Functionally
equivalent — content-addressed, under `.kairos/`, write-once — so this is a
naming difference in the audit brief, not a code defect. No change made.

**Gap found:** nothing previously checked that a stored blob's *current*
bytes still hash to the `sha256` recorded on its `artifacts` row — i.e. bit
rot, an out-of-band edit to `.kairos/content/`, or a `chmod`-defeating manual
write would go undetected forever. `doctor` didn't check this.

**Remediation:** Added a `content_integrity` check to `run_doctor()`
(`kairos.services.doctor`): re-reads every distinct `sha256` referenced by
`artifacts`, recomputes its hash from the stored blob, and reports
mismatched/missing files by id. Added
`tests/integration/test_source_immutability.py`, which:
1. Captures sha256 + mtime of every fixture file used by `scripts/demo.sh`
   before running anything.
2. Runs the full command sequence (init → ingest all six kinds → artifacts →
   show → search → trace → config → logs → note → well → doctor) via
   `CliRunner`.
3. Re-hashes and re-stats every fixture file and asserts byte-for-byte and
   mtime-for-mtime equality with step 1 — proving no command in the surface
   ever mutates a registered source file.
4. Separately corrupts a stored content blob (flips a byte, after
   `chmod`-ing it writable) and asserts `kairos doctor` reports
   `content_integrity` as failed with the affected artifact id named in the
   detail message.

**Verification commands:**
```
pytest tests/integration/test_source_immutability.py -v
pytest tests/unit/test_content_store.py -v
```

## 3. Provenance completeness

| # | Claim | Source | Status |
|---|---|---|---|
| 3.1 | Every result carries artifact id, path, kind, locator, parser name+version, layer | README.md "Why KAIROS"; architecture.md "provenance envelope" | **violated (rendering layer) → remediated** |
| 3.2 | `ProvenanceEnvelope` is the shared shape for every CLI-facing result | architecture.md | **verified** (data model) |

**Evidence of the violation:** `ProvenanceEnvelope`
(`kairos/schemas/provenance.py`) does carry all six fields, and every
service (`search.py`, `show.py`, `trace.py`, `config_query.py`,
`logs_query.py`) builds one correctly. The gap was entirely in the CLI
rendering layer, which is what a user actually sees:

- `cli/commands/search.py` rendered `artifact_id, path, locator, layer,
  snippet` — **missing `artifact_kind`, `parser_name`, `parser_version`**.
- `cli/commands/trace.py` rendered `kind, id, label, artifact_id,
  source_path, locator, layer` (as of the previous hardening pass) — still
  **missing `artifact_kind`, `parser_name`, `parser_version`**.
- `cli/commands/config.py` rendered `locator, layer, source` only —
  **missing `artifact_id`, `artifact_kind`, `parser_name`, `parser_version`**.
- `cli/commands/logs.py` rendered `path, line, timestamp, level, component,
  message` — **missing `artifact_id`, `artifact_kind`, `parser_name`,
  `parser_version`, `layer` entirely**.
- `cli/commands/show.py` renders the full envelope once per artifact (id,
  path, kind, parser+version, parse_status) in the artifact panel, then
  per-span shows `[span_kind] locator layer=X`. This was judged sufficient
  as-is: every span panel is nested under one artifact panel that already
  states id/path/kind/parser once, so repeating all six fields on every
  single span panel would be pure redundant clutter, not new information.
  No change made here.

This is exactly the class of bug the audit warns about: the underlying data
was correct and tested at the schema/service layer, so it would have been
easy to mark this "verified" without checking what the terminal actually
prints. It wasn't.

**Remediation:** Added `kairos/cli/citation.py`, a small shared
citation-rendering module used by every multi-row source-derived command:
- `add_provenance_columns(table, *, include_locator=True)` — adds
  `artifact_id`, `artifact_kind`, `parser`, `layer` (and `locator`) columns
  in a fixed order.
- `provenance_cells(envelope) -> tuple[str, ...]` — renders one envelope as
  matching, already-escaped cell values (`parser` is rendered as `name
  vversion`, e.g. `kairos.markdown v1.0.0`).

`search.py`, `trace.py`, `logs.py`, and `config.py` were rewritten to use
this component instead of hand-building their own column lists, which both
closes the gap and satisfies "create a reusable citation-rendering component
rather than duplicating output formatting across commands." `show.py` was
left as-is per the rationale above (a per-artifact panel is not a per-row
table, so the shared *table* component doesn't apply there, and no
information is actually missing).

**Verification command:**
```
pytest tests/integration/test_provenance_completeness.py -v
```
This new integration test ingests one artifact of each of the six kinds,
then runs `search`, `trace`, `config` (Kconfig only), and `logs` (log only)
and asserts every one of `artifact_id`, `artifact_kind`, the parser name,
and `layer` appears in the captured output — parsing the actual rendered
table text, not the underlying schema, per the audit instruction "add
integration tests that parse CLI output... and assert all required fields
exist."

## 4. Derived-relation discipline

| # | Claim | Source | Status |
|---|---|---|---|
| 4.1 | "Not similarity guesses" — relations are exact, explicit, typed | README.md; architecture.md non-goals | **verified** |
| 4.2 | Every derived relation carries `origin=derived`, a derivation rule, and evidence | architecture.md provenance model | **partially verified → remediated** |
| 4.3 | `trace` differentiates direct match / extraction / derivation / user connection | architecture.md "Trace" section | **partially verified, mostly not-yet-testable** |
| 4.4 | Two lexically-similar-but-distinct terms are never linked | audit requirement | **verified (added adversarial test)** |

**New artifact:** [`docs/relation-registry.md`](relation-registry.md) —
inventories all seven `RelationPredicate` values with subject/object kind,
owning parser, derivation rule string, evidence-span guarantee, determinism
guarantee, and known limitations. Created fresh; nothing like it existed
before.

**4.2 — the hardcoded-layer bug.** Every relation row actually written to
the database today has `origin="derived"` (confirmed by reading all seven
call sites across the six parsers — `HEADING_CONTAINS`, `PAGE_PRECEDES`,
`JSON_CONTAINS`, `MENU_CONTAINS`, `DEPENDS_ON`, `LOG_IN_SESSION`, `IMPORTS`
are all constructed with `origin=Origin.DERIVED`). However,
`kairos.services.trace._build` (the BFS in `trace()`) does not read that
column — it passes the Python literal string `"derived"` directly to
`add_edge(...)` for every relation-table edge, regardless of what's actually
stored. Today these agree by coincidence (100% of relations happen to be
`derived`), but the trace output was not actually deriving its "layer"
column from the source of truth. If any future relation were ever written
with a different origin, `trace` would silently mislabel it.

**Remediation:** `trace.py`'s two `add_edge(...)` calls for relation-table
edges now pass `rel.origin` instead of the literal `"derived"`. This is a
one-line-times-two change; behavior is unchanged today (all existing tests
still pass unmodified) but the value now traces back to the database column
instead of an assumption. Added
`tests/unit/test_relation_registry_consistency.py`, which asserts, for every
parser, that every `Relation` it emits has `origin == Origin.DERIVED`, a
non-empty `derivation_rule`, and (checked against
`docs/relation-registry.md`'s stated evidence guarantee) an
`evidence_span_id` wherever the registry says one is guaranteed. This turns
"the registry is accurate" into something CI actually checks, not just prose.

**4.3 — differentiation is real but was untested, and is partly vacuous
in v0.1.** The edges table already shows `predicate`, `layer`, and
`derivation_rule` per edge, which does let a reader distinguish a `derived`
relation-table edge from an `extracted` mention edge (`"mentioned_in"`,
hardcoded — correctly, since `mentions` has no `origin` column and is
extracted-layer by construction; this one hardcode is *not* a bug). There is
currently no code path in v0.1 that ever writes a `user`-origin `relations`
row (notes and well-membership are the only user-authored tables, and
neither is a `relations` row), so "differentiates a user-authored
connection" is **vacuously true** — there's nothing to mislabel yet. Marked
*not yet testable* rather than *verified*, since there is no fixture that
could fail this check today. No code change; nothing to fix.

**4.4 — adversarial test, added:**
`tests/integration/test_no_false_relations.py` ingests two Markdown files
with headings `"Widget"` and `"Widgets"` (lexically adjacent, textually
distinct) and asserts: (a) they resolve to two separate `entities` rows
(reconciliation is exact-match, case-sensitive, confirmed by reading
`_reconcile_entity`'s use of `find_entities_by_name`, not the fuzzy/CI
variant used only for `trace` seeding), and (b) `kairos trace Widget` never
reaches the `Widgets` entity or its spans via any edge — only via direct FTS
seeding, which is search, not an implied relation, and is not rendered as an
edge.

**Verification commands:**
```
pytest tests/unit/test_relation_registry_consistency.py -v
pytest tests/integration/test_no_false_relations.py -v
```

## 5. FTS correctness

| # | Claim | Source | Status |
|---|---|---|---|
| 5.1 | FTS5 rows map deterministically to `source_spans.id` | architecture.md schema section | **verified** |
| 5.2 | Sync triggers keep the index in lockstep, migration-managed | architecture.md schema section | **verified** |
| 5.3 | A span cannot appear in search after its backing row is gone/invalid | audit requirement | **verified (added test)** |
| 5.4 | `doctor` checks FTS referential consistency | audit requirement (new) | **violated → remediated** |
| 5.5 | Tokenization/ranking behavior is documented | audit requirement (new) | **undocumented → remediated** |

**5.1/5.2 evidence:** `fts.py`'s `search_spans()` always selects and returns
`span_id` (the `UNINDEXED` join column), never the internal FTS `rowid` —
so even though the migration's insert triggers compute the new FTS `rowid`
manually (`COALESCE(MAX(rowid), 0) + 1`), that value is never surfaced or
relied on anywhere; `span_id` is the only identity that matters, and it's
always the real `source_spans.id`. The three triggers
(`source_spans_fts_ai/au/ad`) are created in the one Alembic migration,
confirmed by reading `0001_initial_schema.py`, so no code path can insert a
span without the index following (nothing in the ORM or repositories layer
writes to `source_spans_fts` directly).

**5.3 evidence, now with a regression test:** `search()`
(`kairos/services/search.py`) already does `if span_row is None or
artifact_row is None: continue` for every FTS hit — an orphaned FTS row
(one whose `span_id` doesn't resolve to a live `source_spans` row) is
silently excluded from results rather than crashing or leaking a
dangling reference. This was correct but untested. Added
`tests/integration/test_fts_referential_integrity.py`, which, after a normal
ingest, directly `INSERT`s a bogus row into `source_spans_fts` (a raw SQL
statement — the only way to defeat the trigger-enforced sync, since no ORM
path exists for it) with a `span_id` that doesn't exist, then asserts
`kairos search` still returns only genuine hits and does not crash.

**5.4 — the gap:** nothing previously checked, from the outside, whether
`source_spans` and `source_spans_fts` actually agree in row count or in the
specific set of ids. The triggers *should* guarantee this always, but
`doctor`'s job is exactly to catch the case where something bypassed that
guarantee (manual DB surgery, a future migration that forgets the triggers,
etc.) — which is precisely the scenario 5.3's new test manufactures.

**Remediation:** Added an `fts_consistency` check to `run_doctor()`: counts
`source_spans` vs `source_spans_fts`, and separately counts any `span_id` in
`source_spans_fts` with no matching `source_spans.id` (a `LEFT JOIN ...
WHERE source_spans.id IS NULL` via raw SQL, since the virtual table has no
ORM model). Reports both counts and flags a mismatch as failed. The same
`test_fts_referential_integrity.py` test also asserts `kairos doctor`
reports this check as failed once the bogus row is injected, and healthy
before.

**5.5 — documented, not previously written down anywhere:** Added a short
paragraph to `docs/architecture.md`'s schema section stating that
`source_spans_fts` uses FTS5's default `unicode61` tokenizer (case-folding,
Unicode-aware word splitting, no stemming) and that ranking is `bm25()` with
FTS5's default column weights (a single indexed column, so weighting is
moot). This was true of the existing implementation already — no behavior
changed, only documented.

**Verification commands:**
```
pytest tests/integration/test_fts_referential_integrity.py -v
kairos doctor   # run inside any workspace; see fts_consistency and content_integrity rows
```

## 6. Parser boundaries and malformed input

| # | Claim | Source | Status |
|---|---|---|---|
| 6.1 | Every parser: well-formed + malformed coverage, correct `parse_status`, no unhandled traceback | architecture.md parser registry; implementation-plan.md test plan | **partially verified → remediated** |
| 6.2 | Kconfig: unknown fields remain available in raw metadata | audit requirement (new, stricter than prior docs) | **violated → remediated** |
| 6.3 | Kconfig: `kairos config` on a missing symbol is actionable | cli.md | **verified** (existing test) |
| 6.4 | PDF: extraction failure vs. legitimate image-only page are distinguished | audit requirement | **verified** (already true, previously untested at this granularity) |

**6.1 — two concrete gaps found and fixed, both about "never silently
discard," the project's own founding principle:**

1. **`repository_files.py` (Python AST parser):** module, class, and
   function entities were created with `origin=Origin.EXTRACTED` but *no
   `Mention` row* linking each entity to the span it was extracted from —
   unlike every other parser that creates entities (Markdown headings,
   Kconfig symbols, log sessions all create a grounding `Mention`). An
   extracted-layer entity with no mention has no evidence trail: `kairos
   trace ClassName` would find the entity but the node it displays would
   have empty provenance, structurally indistinguishable from an entity that
   was never grounded in a source at all. Fixed: `PythonParser` now emits a
   `Mention` for the module entity, and for every class/function entity, to
   its own defining span. Added
   `tests/unit/test_parsers_repository.py::test_entities_are_grounded_by_mentions`
   asserting `len(result.mentions) == ` (module + every class + every
   function entity), mirroring the equivalent Kconfig test
   (`test_symbols_become_entities_with_mentions`) that already enforced this
   invariant for a different parser.

2. **`logs.py` (log parser):** a session-boundary line (`=== BOOT ===`)
   became a `log_session` entity but — by explicit prior design, asserted in
   `test_all_lines_produce_log_line_spans` with the comment "boundary line
   itself is not a span" — got no span and therefore no mention either. The
   session entity had zero grounding evidence anywhere. This broke the same
   invariant as (1), and contradicted the pattern established by every other
   entity-creating parser in this same codebase. Fixed: the boundary line
   now gets its own `LOG_LINE` span (metadata-tagged
   `is_session_boundary: true`) and a `Mention` from the session entity to
   that span. The previous test's assumption was updated (`len(result.spans)
   == 5`, not 4) with its comment corrected; `log_in_session` relation count
   is unchanged (still only lines *after* the boundary, not the boundary
   itself, matching the existing "session contains the lines within it"
   semantics).

Neither of these produced an unhandled traceback in practice — both are
provenance-completeness gaps (undocumented implementation vs. the
project's own established pattern), not crashes.

**Recursion depth (documented risk, not remediated):** `json_parser.py`'s
`visit()` and `kconfig.py`'s `visit()` both recurse per nesting level with
no depth guard. A deliberately adversarial, deeply-nested JSON/Kconfig
document (thousands of levels) could raise `RecursionError`, which is not a
`KairosError` and would currently surface as an uncaught exception (now
mapped to exit code 3 by the CLI failure-contract fix in §7, rather than a
bare traceback — see below). Converting these to iterative stack-based
walks would close this fully but touches three parsers' core traversal
logic for a threat model (adversarial nesting depth, not malformed real-world
input) outside this audit's mandate to avoid scope expansion. Documented
here as a known limitation; no code change.

**6.2 — Kconfig unknown-field preservation, the gap:** `KconfigParser`'s
`metadata` dict for each span hardcodes exactly six keys (`node_type`,
`prompt`, `symbol_type`, `default`, `depends_on`, `choices`) — any other key
present on a Kconfig node in the source JSON (e.g. `help`, `range`, a
project-specific extension field) was read from the raw bytes (preserved
untouched in the content store) but never surfaced anywhere in the
*structured* view (`source_spans.metadata_json`), meaning `kairos show` and
`kairos config` could never display it. This is not "silently discarding
the source" in the strict sense (raw bytes are intact and `kairos show
--locator` still exists) but it does contradict "verify unknown fields
remain available in raw metadata" as a specific, stricter requirement this
audit imposes. **Fixed:** every node key not in the known set is now
collected into `metadata["extra"]` (a `dict[str, object]`, empty when there
are no extra keys). Added
`tests/unit/test_parsers_kconfig.py::test_unknown_fields_preserved_in_extra_metadata`,
which ingests a symbol carrying a `"help"` field not in the known schema and
asserts it surfaces at `span.metadata["extra"]["help"]`.

Also found and fixed in the same file, same category: a `children` list
entry that is not a JSON object (e.g. a stray string or number in the array)
was silently `isinstance`-filtered out with **no diagnostic at all** —
`elif isinstance(child, dict): visit(...)`, with no `else`. This is a real
"silently discard malformed content" violation. Fixed: non-dict children now
append a `Diagnostic` naming the parent menu path and the malformed child's
value, and `parse_status` downgrades to `PARTIAL`. Added
`tests/unit/test_parsers_kconfig.py::test_non_dict_child_is_diagnosed_not_dropped`.

**6.3 — already verified**, no change:
`test_config_unknown_symbol_fails_cleanly` (integration) already asserts a
non-zero, actionable exit for a nonexistent symbol via
`ConfigSymbolNotFoundError`.

**6.4 — already true, now explicitly tested:** `pdf.py` already emits two
*different* diagnostic messages — `"Could not extract text from page N:
{exc}"` (an exception during extraction — a real extraction failure) versus
`"Page N has no extractable text"` (extraction succeeded and returned
empty/whitespace — the legitimate image-only-page case). This distinction
existed and was correct but had no test asserting the *message* differs
between the two cases (the existing `test_blank_page_is_recorded_not_dropped`
only exercises the second case). Added an assertion to
`test_parsers_pdf.py` covering a page whose `extract_text()` raises,
confirming the message says "Could not extract text from," distinct from
the no-text-content message.

**Verification commands:**
```
pytest tests/unit/test_parsers_repository.py -v
pytest tests/unit/test_parsers_logs.py -v
pytest tests/unit/test_parsers_kconfig.py -v
pytest tests/unit/test_parsers_pdf.py -v
```

## 7. CLI failure contract

| # | Claim | Source | Status |
|---|---|---|---|
| 7.1 | "Every command returns a non-zero exit code on failure, with a message... prefixed Error:" | cli.md; README.md | **verified for the code≠0 part; the taxonomy didn't exist** |
| 7.2 | Standardized exit codes (0/1/2/3) | audit requirement (new — not a prior product claim) | **violated → remediated** |
| 7.3 | Tracebacks suppressed by default, preserved only under an explicit debug mechanism | audit requirement (new) | **violated → remediated** |

**7.1 evidence:** true today, verified by seven existing integration tests
across artifact/well/note/search/doctor/config command families, each
asserting `exit_code != 0` (in fact always `== 1`) plus an `"Error"` string
in output for a bad artifact id, a nonexistent ingest path, a malformed FTS5
query, a missing workspace, a duplicate well name, an unknown Kconfig
symbol, and an unknown note/well target. Every one of these passed before
this audit and continues to pass after — but see 7.2: they all collapsed
onto the single exit code 1, with no distinction between "you gave bad
input" and "your workspace itself is broken."

**7.2 — the gap:** `KairosError` had no severity/category concept;
`cli/errors.py`'s `cli_command` decorator hardcoded `typer.Exit(code=1)` for
every `KairosError` subclass, and `cli/commands/doctor.py` separately
hardcoded its own `typer.Exit(code=1)` on a failed health check. There was
no code path that ever produced exit code 2 or 3, despite this audit's
required taxonomy (0 success / 1 expected user input or domain failure / 2
workspace-configuration-or-integrity failure / 3 unexpected internal error).

**Remediation:**
- `KairosError` gained a class attribute `exit_code: int = 1`.
  `WorkspaceNotFoundError`, `WorkspaceAlreadyExistsError`, and
  `ReadOnlySourceViolationError` override it to `2` (workspace/configuration/
  integrity failures — running outside a workspace, re-initializing an
  existing one, or a read-only-boundary violation are all "your environment
  is wrong," not "you typed a bad id"). Every other existing subclass
  (`ArtifactNotFoundError`, `SourcePathNotFoundError`, `InvalidLocatorError`,
  `InvalidSearchQueryError`, `NoParserAvailableError`, `WellNotFoundError`,
  `WellAlreadyExistsError`, `WellMemberNotFoundError`,
  `TargetNotFoundError`, `ConfigSymbolNotFoundError`) keeps the default `1`
  — all of these are "you asked for something that doesn't exist or isn't
  valid," squarely user/input/domain failures.
- `cli/errors.py`'s decorator now raises `typer.Exit(code=exc.exit_code)`
  instead of a hardcoded `1`.
- `cli/commands/doctor.py` now raises `typer.Exit(code=2)` on any failed
  check (a failing `doctor` check is definitionally a
  workspace/integrity-category failure, never a user-input error).
- Updated the two existing tests that asserted the old, now-wrong exit code:
  `test_commands_require_a_workspace` and `test_doctor_requires_a_workspace`
  now assert `exit_code == 2`. Every other existing exit-code assertion
  (all the code-1 cases) required no change, since none of those error
  classes moved.
- Added `tests/integration/test_exit_code_taxonomy.py`, one test per
  category, asserting the *exact* code (not just non-zero) for: a missing
  workspace (2), `init` on an existing workspace (2), a failing `doctor`
  check on a workspace with a hand-corrupted content store (2), a bad
  artifact id / bad locator / unknown Kconfig symbol / unknown well /
  invalid well operation (1 each), and an unexpected internal error (3, see
  7.3).

**7.3 — the gap:** `cli_command` only ever caught `KairosError`; any other
exception (a real bug — `AttributeError`, `RecursionError`, an
unparameterized SQL error surfacing as a raw `sqlalchemy` exception, etc.)
propagated uncaught. Typer/Click's default handler then rendered a full
Rich-formatted traceback to the user on every invocation, unconditionally —
there was no way to suppress it, and no `--debug` mechanism existed to make
the distinction meaningful even if there had been.

**Remediation:** `cli_command` now also catches bare `Exception`. By
default it prints a single actionable line — `Error: an unexpected internal
error occurred: {summary}. Set KAIROS_DEBUG=1 to see the full traceback.` —
and exits with code `3`. If the environment variable `KAIROS_DEBUG` is set
(to any non-empty value), the exception is re-raised unchanged, giving the
full traceback exactly as before. This is the smallest possible "explicit
debug mechanism": an env var, not a new global Typer option threaded through
eleven command modules, since no such option existed previously and the
audit only requires *some* explicit opt-in, not a specific flag shape.
Documented in `docs/cli.md`.

**Verification command:**
```
pytest tests/integration/test_exit_code_taxonomy.py -v
KAIROS_DEBUG=1 kairos show not-a-real-id   # compare traceback presence with/without the env var
```

## 8. Architecture boundary enforcement

| # | Claim | Source | Status |
|---|---|---|---|
| 8.1 | `domain -> infrastructure -> services -> cli`, domain has zero framework imports | architecture.md; CONTRIBUTING.md "Architecture rules" | **verified, previously only by convention → now enforced in CI** |

**Evidence:** manual inspection of every file in `src/kairos/domain/`
(`enums.py`, `errors.py`, `ids.py`, `json_types.py`, `locators.py`,
`models.py`, `parser.py`) confirms zero imports of `typer`, `rich`,
`sqlalchemy`, `alembic`, `pypdf`, or any `kairos.infrastructure`/`kairos.cli`
module — only stdlib (`dataclasses`, `enum`, `re`, `uuid`, `pathlib`,
`typing`) and intra-domain imports. This was true, but enforced by nothing
except a human reading the code, per CONTRIBUTING.md's own admission
("enforced by convention, not tooling").

**Remediation:** Added
`tests/unit/test_architecture_boundaries.py`. It does not introduce a new
linting framework — it's a single pytest test that walks every `.py` file
under `src/kairos/domain/`, parses it with `ast` (already a stdlib
dependency used elsewhere in this codebase), collects every `import X` /
`from X import ...` at module level, and asserts none of them start with any
of `typer`, `rich`, `sqlalchemy`, `alembic`, `pypdf`, `kairos.infrastructure`,
`kairos.cli`, `kairos.services`, `kairos.schemas`. Since this test runs as
part of the existing `pytest -q` step already gated in
`.github/workflows/ci.yml`, CI now fails automatically if a future change
introduces a prohibited import into `domain/` — satisfying "CI fails if a
prohibited layer import is added" without adding new tooling (no import-linter
package, no separate CI job).

**Verification command:**
```
pytest tests/unit/test_architecture_boundaries.py -v
```

## 9. Reproducibility and clean-room setup

| # | Claim | Source | Status |
|---|---|---|---|
| 9.1 | A fresh clone, venv, install, test, and demo succeed offline | README.md installation instructions; CONTRIBUTING.md dev setup | **partially verified → remediated (CI now runs the demo)** |
| 9.2 | Dependencies are appropriately constrained for reproducible CI | audit requirement | **verified** (existing ranges judged adequate) |
| 9.3 | Tests don't depend on the developer's home directory, locale, or network | audit requirement | **verified** |

**9.1 — the gap:** `.github/workflows/ci.yml` ran `ruff format --check`,
`ruff check`, `pyright`, and `pytest -q` — but never `scripts/demo.sh`. The
demo script existed, was portable (its own `PYTHON_BIN` autodetection loop
and `mktemp -d` workspace, no hardcoded paths), and had been manually
verified to run cleanly in earlier work on this project, but nothing
re-verified that on every push/PR the way the rest of the quality gate does.
**Remediation:** added a `Demo` step to `ci.yml`, running
`bash scripts/demo.sh` with `KAIROS_BIN` pointed at the just-installed
console-script entry point, after the `pytest` step. A regression in any
command's basic behavior across all six parser kinds now fails CI, not just
a future manual run.

**9.2:** `pyproject.toml` already pins every direct dependency to an
upper-bounded range (`typer>=0.12,<1.0`, `rich>=13.7,<14.0`,
`pydantic>=2.7,<3.0`, `sqlalchemy>=2.0,<3.0`, `alembic>=1.13,<2.0`,
`pypdf>=4.2,<5.0`). This is judged adequate for a v0.1 tool of this size —
adding a full lockfile (`pip-tools`/`uv.lock`) would be new tooling and
process overhead disproportionate to what this audit calls for. No change
made; documented as a deliberate choice, not an oversight.

**9.3:** confirmed by reading every test file — all use `tmp_path`
(pytest's built-in isolated temp-directory fixture) for any workspace state,
`FIXTURES = Path(__file__).parent.parent / "fixtures"` for input data (never
an absolute path outside the repo), and no test reads `$HOME`,
`os.path.expanduser`, or does locale-sensitive string formatting (dates are
handled via explicit `datetime`/ISO formatting, not locale-dependent
`strftime` directives). No network calls in any test (confirmed by the same
import grep as §1, extended to `tests/`).

**Verification commands:**
```

pip install -e ".[dev]"
pytest -q
bash scripts/demo.sh
```
