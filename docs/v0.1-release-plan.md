# KAIROS v0.1.0 — Release Plan

Written before any release-hygiene code changes, per the release-engineering
workflow. Grounded in an actual trial build (`python -m build --no-isolation`
against this checkout), not assumptions — findings below are what that build
produced, inspected file-by-file.

## 1. Existing release path

- **Version**: `pyproject.toml`'s `[project] version = "0.1.0"` already
  matches the target release. `src/kairos/__init__.py` independently
  hardcodes `__version__ = "0.1.0"` — a duplicated mutable string, not
  currently a mismatch, but two places that could drift on the next release
  if only one gets bumped.
- **No runtime version display**: there is no `kairos --version` flag today.
  Nothing depends on `__version__` at runtime (confirmed by grep — the only
  reference to it is its own definition).
- **Build backend**: plain `setuptools.build_meta` via
  `[build-system] requires = ["setuptools>=68", "wheel"]`. No
  `setuptools-scm`, no `hatchling`, no dynamic versioning — this is already
  about as simple as packaging gets, and the task's "do not add a complex
  release framework" constraint is satisfied by leaving it exactly as is.
- **CI**: `.github/workflows/ci.yml` runs `ruff format --check`,
  `ruff check`, `pyright`, `pytest -q`, and a `scripts/demo.sh` smoke test
  (added in the prior audit) on every push/PR to `main`. It does not yet
  build or install distribution artifacts.
- **Demo**: `scripts/demo.sh` already exercises init → ingest (all six
  parser kinds) → artifacts → search → show → trace → config → logs →
  note add/list → well create/add/show/list → search --well → doctor
  against synthetic fixtures under `tests/fixtures/`, with `set -euo
  pipefail` (fails loudly on any non-zero command). It assumes Bash.

## 2. Gaps found (trial build evidence)

1. **Duplicated version string** (`pyproject.toml` + `__init__.py`) — no
   single source of truth enforced.
2. **No `--version` CLI flag** — a reviewer verifying an installed artifact
   has no fast way to confirm which version they're running without
   inspecting package metadata by hand.
3. **No `CHANGELOG.md`** at all.
4. **No build verification** — nothing builds or inspects `sdist`/`wheel` in
   CI; a packaging regression (e.g. a file silently excluded) would only be
   caught by a human running `python -m build` manually.
5. **Package-data gap, confirmed by inspection**: the trial wheel and sdist
   both include `kairos/infrastructure/database/migrations/env.py` and
   `.../versions/0001_initial_schema.py` but **not**
   `.../migrations/script.py.mako` — setuptools only auto-includes `.py`
   files inside discovered packages; a `.mako` template needs an explicit
   `package-data` entry. Functionally harmless today: `alembic.command.
   upgrade()` (what `kairos init` actually calls) only reads `env.py` and
   `versions/`; `script.py.mako` is only consulted by `alembic revision`,
   a development-time command never invoked by KAIROS itself. Not a
   runtime bug, but worth including for packaging completeness at near-zero
   cost.
6. **No artifact-install verification** — nothing proves the *built*
   wheel/sdist, installed into a clean environment with no repo checkout on
   `sys.path`, actually runs. This is the gap most directly tied to this
   task's definition of done.
7. **`demo.sh` doesn't clean up its own workspace** — it deliberately prints
   `Workspace left at: $WORKSPACE` for manual inspection. Fine for a
   developer convenience script, not fine for a release smoke test that's
   supposed to leave no residue and end with an unambiguous success message.
8. **`demo.sh` assumes Bash** with no documented Windows-native alternative
   beyond "install Git Bash" (which is what this environment already uses,
   and what GitHub Actions' `ubuntu-latest` runner provides natively — both
   are real, tested environments, not assumed ones).
9. **`docs/v0.1-status.md`'s "Audit verification" section** (added in the
   prior audit) lists test paths but was never extended to cover
   distribution-artifact verification, and its expected test count needs
   re-confirming against the actual final count before tagging.
10. **No annotated tag or GitHub Release exists yet.**

Nothing found requires a product-behavior change — every gap is packaging,
CI, documentation, or process hygiene, consistent with "release engineering
only."

## 3. Verification commands and expected outcomes

| Command | Expected outcome |
|---|---|
| `pytest -q` | All tests pass; exact count confirmed empirically before tagging (see §7) |
| `ruff format --check .` | No files would be reformatted |
| `ruff check .` | No lint violations |
| `pyright` | 0 errors, 0 warnings |
| `python -m build` | Produces exactly one sdist and one wheel in `dist/`, both containing every runtime file and no dev/test/workspace/editor artifacts |
| Artifact-install script (new, §4 below) | Fresh venv, wheel-only install, `kairos --version` resolves, init/ingest/search/show/trace/doctor all exit 0, using an interpreter whose `sys.path` cannot reach `src/` |
| `bash scripts/demo.sh` | Exits 0, prints a final success message, leaves no temp directory behind |

## 4. Minimal commit plan

Small, logically grouped commits, each independently green:

1. Version single-source-of-truth + `--version` CLI flag + `CHANGELOG.md`.
2. Packaging hygiene: `script.py.mako` package-data fix; confirm build
   excludes are correct (no code change expected here beyond the
   package-data entry — this step is mostly verification).
3. `scripts/verify_install.sh` (new): build → fresh venv → install wheel →
   run offline workflow → assert exit codes → assert the installed module
   resolves outside the repo checkout.
4. `scripts/demo.sh` hardening: auto-cleanup via `trap`, explicit success
   message, brief top-of-file note on the Bash requirement.
5. CI: add build + artifact-install-verification steps to
   `.github/workflows/ci.yml`.
6. Documentation: real "Audit verification" additions to
   `docs/v0.1-status.md` (distribution verification subsection, re-run test
   count), a short "Release verification" pointer in `README.md`, this
   plan's own findings reconciled once implemented.
7. Tag `v0.1.0` and publish the GitHub Release, once everything above is
   green on `main`.

No step depends on inventing new product behavior; every step is either a
build-metadata correction, a new verification script/CI job, or a
documentation update reflecting what was actually run.
