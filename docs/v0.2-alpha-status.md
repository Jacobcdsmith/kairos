# KAIROS v0.2-alpha Status — Terminal Lineage Interface

This milestone adds `kairos tui`, a Textual-based presentation layer over
the v0.1 substrate documented in [docs/v0.1-status.md](v0.1-status.md).
v0.1's contract is unchanged; nothing in this milestone touches domain,
infrastructure, or the schema.

## Completed

- **`kairos tui`** launches a full-screen, three-pane (Explorer /
  Workspace / Evidence) Textual application with a persistent `:command`
  line and status line, backed entirely by existing `kairos.services.*`
  functions through `kairos.tui.controller.dispatch_text`.
- **Optional dependency, cleanly isolated**: Textual is behind
  `pip install -e ".[tui]"`. `kairos tui` without it prints an install
  message and exits 1, never an import traceback. Enforced by
  `tests/unit/test_architecture_boundaries.py::test_only_kairos_tui_imports_textual`
  and `tests/tui/test_optional_dependency.py`.
- **Command grammar**: `:home`, `:artifacts [kind]`, `:search`, `:show`,
  `:trace`, `:well list|use|clear|show`, `:config`, `:logs`, `:doctor`,
  `:note list|add`, `:history`, `:help`/`?`, `:refresh`/`r`, `:quit`/`:q`,
  plus aliases `:s`, `:t`. Unknown commands produce an actionable error
  naming the closest valid command, never a traceback.
- **Full citation everywhere**: every rendered search hit, trace node,
  span, log line, and config lookup shows its complete
  `ProvenanceEnvelope` in the Evidence pane — artifact id, path, locator,
  parser+version, and layer — matching the CLI's contract exactly.
- **Active coherence well as visible context boundary**: the header
  always shows `well: none` or `well: <name>`; `:search`/`:trace` honor
  it via the same `well_artifact_ids` scoping the CLI uses.
- **Responsive layout**: >=120 cols shows all three panes; 80-119 hides
  Evidence from the default view (still Tab-reachable); <80 hides
  Explorer and Evidence, keeping the Workspace transcript's citations
  fully unwrapped rather than truncated.
- **One new service, minimally scoped**: `kairos.services.activity.recent_events`,
  a thin read wrapper around the existing `list_events` repository
  function — the only service-layer gap found during implementation (see
  [docs/tli-implementation-plan.md](tli-implementation-plan.md)).
- **Async, non-blocking**: every service call runs in a Textual thread
  worker (`@work(thread=True, exclusive=True)`); a monotonically
  increasing request id discards stale results if a newer command
  supersedes an in-flight one.
- **Read-only enforcement**: the controller's mutation surface is exactly
  `add_note` plus well activate/clear — no `create_well`, `add_member`,
  ingest, or doctor-repair action exists anywhere in the TUI.
- **Copy citation/excerpt (`c`/`y`)**: uses Textual's own OSC 52 clipboard
  write (never a shell command) and always also echoes the plain text
  into the Workspace transcript, since an unsupported terminal silently
  no-ops the clipboard write with no signal otherwise.
- **CI updated**: `.github/workflows/ci.yml` installs `.[tui,tui-test]`
  and runs `pytest -q tests/tui` as a separate step after the existing
  gate, so the TUI's interactive surface (including all 22 Pilot tests)
  actually runs in CI, without changing the base job's `.[dev]`-only
  install or its own `pytest -q` step.

## Test count

- v0.1 baseline: 99 passing (`pytest -q` with only `.[dev]` installed).
- v0.2-alpha adds one new architecture-boundary assertion
  (`test_only_kairos_tui_imports_textual`) plus `tests/tui/`: 3
  optional-dependency tests, 16 pure-Python controller tests, and 22
  Pilot-driven app tests.
- Full suite with `.[dev,tui,tui-test]` installed:
  **141 passing** (99 + 1 + 3 + 16 + 22), confirmed by running
  `pytest -q` end to end.
- With only `.[dev]` installed (no Textual, no pytest-asyncio) —
  empirically verified, not just asserted: `tests/tui/test_app.py`
  cleanly **skips** (via `pytest.importorskip`) instead of erroring or
  silently no-op-passing (a real risk with bare `async def` tests and no
  asyncio plugin); `tests/tui/test_controller.py` and
  `tests/tui/test_optional_dependency.py` still run and pass (119 total),
  since neither needs Textual.

Reproduce:

```bash
pip install -e ".[dev]"                 # v0.1 baseline, 99 passing
pip install -e ".[dev,tui,tui-test]"    # adds the TUI + its test runner
pytest -q                                # full suite, exact count above
bash scripts/demo.sh                     # v0.1 CLI smoke test, unchanged
```

## Known alpha limitations

- **`:logs` and `:config` are not well-scoped.** The underlying services
  (`query_logs`, `get_config_symbol`) don't take a `well` parameter; the
  TUI does not fake this — an active well only narrows `:search`/`:trace`.
- **Well creation and membership changes stay CLI-only.** The `w` picker
  can only activate or clear an existing well.
- **`doctor` has no repair action** in this interface, by design.
- **History replay is view-only.** `Ctrl+R` shows past commands and their
  status; re-running a specific historical entry isn't wired up yet
  (`:refresh`/`r` re-runs only the most recent successful command).
- **Narrow-width (<80 col) layout** hides Explorer/Evidence outright
  rather than offering per-pane full-screen navigation screens; reachable
  via Tab-cycling instead.
- **No `Left`/`Right` hierarchical drill-down.** The Explorer is a flat
  list per mode; there is no "move to parent" / "expand child" navigation
  beyond what `Up`/`Down` plus re-issuing a command (`:show`, `:trace`)
  already provides. Not in `:help` or `docs/tli.md`'s keybinding table —
  removed rather than left advertised-but-unbound.
- **`TuiState` deviates from an initial spec sketch** in two documented
  ways: the active well is stored by name (matching every service
  signature) rather than by id, and `:note` commands get their own
  `"notes"` mode rather than overloading `"show"` with an incompatible
  result shape. Both are explained in
  [docs/tli-implementation-plan.md](tli-implementation-plan.md).

## Confirmation

No LLM integration, network access, telemetry, embeddings/vector/fuzzy
similarity, autonomous execution, background watcher, shell execution,
source-file mutation, multi-agent behavior, or web/HTTP/WebSocket surface
was introduced by this milestone. `kairos.tui` is read-mostly Textual
presentation code over the unchanged v0.1 service layer.
