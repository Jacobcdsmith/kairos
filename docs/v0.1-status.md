# KAIROS v0.1 Status

## Completed

- **Workspace**: `kairos init` creates `.kairos/` (SQLite DB, content
  store, JSONL event log), runs the schema migration.
- **Schema**: all nine specified tables, plus an FTS5 virtual table and
  three sync triggers, in one Alembic migration.
- **Content store**: SHA-256-addressed, write-once, chmod read-only.
- **Six first-class parsers**, each producing spans + entities + mentions +
  relations + diagnostics, never silently dropping malformed input:
  plain text/Markdown, PDF, JSON, Kconfig-menu JSON, runtime/emulator logs,
  Python repository files (AST-based).
- **Provenance envelope**: every search hit, trace node/edge, and `show`
  span carries artifact id, workspace-relative path, kind, an exact
  locator (discriminated union, one variant per source kind, with a
  round-trippable string form), parser name+version, and its
  raw/extracted/derived/user layer.
- **Cross-artifact entity reconciliation**: parser-emitted entities dedup
  by `(canonical_name, entity_type)` at ingest time, which is what lets
  `trace` cross document/file boundaries (verified by an integration test:
  two Markdown files sharing a heading, traced from a bare word in one
  document's paragraph to a sibling span in the other, in two hops).
- **Full CLI**: `init`, `ingest`, `artifacts`, `show`, `search`, `trace`,
  `note add/list`, `well create/add/remove/show/list`, `config`, `logs`,
  `doctor`. Every command exits non-zero with an actionable message on
  failure (verified for: unknown artifact/well/target ids, a nonexistent
  ingest path, a malformed FTS5 query, running outside a workspace,
  duplicate well names).
- **Event model**: every mutating command appends to both the `events`
  table and `.kairos/events.jsonl`.
- **Tests**: 96 passing — unit tests per parser (including a malformed
  variant of each fixture) plus locator/content-store unit tests, and CLI
  integration tests (via Typer's `CliRunner`) covering every command,
  every parser kind, well-scoped search, and the cross-artifact trace
  scenario above. Extended by the implementation audit (see below) with
  network-blocked, source-immutability, provenance-rendering, relation-
  discipline, FTS-corruption, exit-code, and architecture-boundary
  regression tests.
- **Quality gate**: `ruff format`/`ruff check` clean, `pyright --strict`
  clean (0 errors) across `src` and `tests`.

## Audit verification

initial v0.1 build — see [docs/v0.1-audit.md](v0.1-audit.md) for the
per-claim findings and [docs/relation-registry.md](relation-registry.md) for

where the original claim was too strong to honestly keep.

## Known limitations

- **Kconfig `depends_on`** is only decomposed into `depends_on` relations
  for a simple identifier or a `&&` conjunction of identifiers. Anything
  with `||`, `!`, `=`, or parentheses is kept verbatim on the symbol's span
  metadata (not silently discarded) but doesn't produce a graph edge.
- **Python import resolution** is name-based, not path-based: `import foo`
  links to whichever ingested module has `canonical_name == "foo"`,
  regardless of where it lives. This is deliberately consistent with how
  Markdown headings resolve, but it means two unrelated `foo.py` files in
  different, unrelated ingests will be treated as the same import target.
- **Class/function entity names are bare, not fully qualified** (`Widget`,
  not `sample.Widget`) — the same tradeoff as above, applied consistently.
- **No concurrent-writer story.** SQLite's default locking is relied on;
  v0.1 assumes one owner running one CLI invocation at a time against a
  workspace.
- **No workspace-directory move/rename support.** `original_path` is
  stored as an absolute path at ingest time; moving the workspace or the
  ingested source afterward doesn't break the content-addressed store, but
  a `source_path` computed against the old location will no longer be
  workspace-relative.
- **`well remove` takes the membership id**, not the target's artifact/span
  id (visible via `well show`) — a specific-enough choice to review if it
  turns out to be a rough edge in practice.
- **Log line pattern** is fixed (`<timestamp> <LEVEL> [component]
  message`), not configurable per-ingest in v0.1.

## Stable contracts (safe to build on)

- The nine schema table names and column names, exactly as specified.
- `ProvenanceEnvelope`'s shape and the six `locator_kind` values and their
  string forms.
- The `Origin` vocabulary (`raw`, `extracted`, `derived`, `user`, and the
  reserved-but-unused `model`).
- The `Parser` protocol (`kind`, `parser_name`, `parser_version`, `sniff`,
  `parse` -> `ParseResult`).
- `.kairos/events.jsonl`'s line shape (`id`, `occurred_at`, `event_type`,
  `payload`).
- Every CLI command's exit-code contract: `0` success, `1` expected user/
  input/domain failure, `2` workspace/configuration/integrity failure, `3`
  unexpected internal error (traceback suppressed unless `KAIROS_DEBUG=1`)
  — see [docs/cli.md](cli.md#exit-codes).

## Next milestone

Framework-substrate work that stays within the local-terminal-application
scope of this milestone, still without model inference or model-provider
integration:

- A configurable log-line pattern per ingest (rather than one fixed
  regex), so `kairos logs` covers log formats beyond the built-in one.
- Path-aware (not just name-based) Python import resolution, using the
  ingested directory tree to disambiguate same-named modules.
- A richer Kconfig `depends_on`/`select` expression parser (boolean
  connectives beyond `&&`), still refusing to guess at anything it can't
  parse exactly.
- `well remove` accepting either the membership id or a target id directly
  (resolving ambiguity when a target is in a well more than once, which
  can't currently happen but would need a decision if `well add` ever
  allowed duplicates).
- A `kairos events` command to query the event log directly from the CLI
  (today it's inspectable via `.kairos/events.jsonl` or SQL, but has no
  dedicated command).
- Bounded, permissioned local actions on top of this substrate — the
  "later" mentioned in the product thesis — remain explicitly out of
  scope until a future milestone defines their permission model.
